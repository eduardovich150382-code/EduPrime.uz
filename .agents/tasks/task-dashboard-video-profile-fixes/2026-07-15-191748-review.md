# YouTube player simplification, live dashboard stats, and profile management

Three features in one diff: the YouTube embed component was gutted of its broken IFrame API approach and replaced with a simple `youtube-nocookie.com/embed` iframe using native controls, fixing Error 153. The dashboard was wired to a new `/api/dashboard/stats` endpoint that computes real test count, average score, rank, and streak from the database. A full profile page was built with avatar upload (UploadThing), name editing, and connected-account display for Google and Telegram.

**Watch for:** Debug endpoint shipped to production exposes internal DB structure (confirmed). Profile image URL is accepted from the client with no domain validation, allowing arbitrary external images to be stored as the user avatar (confirmed). Rank computation loads all users' aggregated scores into application memory on every dashboard load (confirmed). UploadThing avatar endpoint has no auth middleware, so unauthenticated users can upload files (likely).

## High-level view

The old YouTube component tried to use the IFrame API with `sandbox` restrictions and `controls=0`, a combination YouTube rejects with Error 153. The replacement embeds via `youtube-nocookie.com` with native controls and no sandbox attribute.

The dashboard stats endpoint performs five sequential database queries per request. The rank calculation uses `groupBy` across all users and filters in JS memory, which will degrade as the user table grows.

The profile API accepts a `PATCH` with `{ name, image }` where `image` is any string that passes a truthy/trimmed check. There is no validation that the URL comes from UploadThing, so a client can set their avatar to any external URL. The UploadThing `avatar` FileRouter endpoint itself has no `.middleware()` call, meaning file uploads are not gated by authentication at the UploadThing layer.

A `/api/debug` route was added that exposes teacher records, all published tests, all subjects, and internal query results to any authenticated user. This should not ship.

1. **Debug endpoint in production** — `/api/debug/route.ts` exposes internal database records (teacher, tests, subjects) to any authenticated user. Remove before shipping.
2. **No image URL validation on profile PATCH** — The `image` field in `PATCH /api/profile` accepts any string. Validate that it matches the UploadThing domain (`utfs.io`) to prevent storing arbitrary external URLs.
3. **UploadThing avatar endpoint unauthenticated** — The `avatar` FileRouter has no `.middleware()` auth check. Add a middleware that verifies the session before accepting uploads.
4. **Rank query scales poorly** — `groupBy` on all `testResult` rows followed by in-memory filtering will slow down as the user base grows. Replace with a single SQL query using a subquery or window function, or cache rank periodically.
5. **console.log statements in production API routes** — `teacher/tests/route.ts` has verbose `console.log` calls that leak internal IDs to server logs. Remove or gate behind a debug flag.

## Debug endpoint shipped to production

`src/app/api/debug/route.ts` returns teacher records, all published tests, all subjects, and the full result of the teacher-tests query including any Prisma errors — all behind nothing more than a session check. Any logged-in user (not just admins or teachers) can hit this endpoint and see the entire test catalog with subject and teacher ID mappings. The file has a comment "Temporary debug endpoint — remove after fixing" which confirms it's scaffolding, but it's included in the diff being shipped.

## Profile image URL injection

The `PATCH /api/profile` handler:

```typescript
if (typeof image === 'string' && image.trim().length > 0) {
  updateData.image = image.trim();
}
```

This stores any URL into the user's `image` field. A malicious client can set their avatar to a tracking pixel, an offensive image hosted externally, or a URL that later 404s. The intended flow is: upload via UploadThing -> get back a `utfs.io` URL -> send that to the PATCH endpoint. But nothing enforces this flow server-side. Validate that `image` starts with the expected UploadThing domain prefix.

## UploadThing avatar endpoint auth gap

The `avatar` entry in `ourFileRouter` (in `core.ts`) uses only:

```typescript
avatar: f({
  image: { maxFileSize: '1MB', maxFileCount: 1 },
}).onUploadComplete(async ({ file }) => {
  return { url: file.url };
});
```

There is no `.middleware()` call chained before `.onUploadComplete()`. In UploadThing's API, the middleware is where you verify the user's session and reject unauthenticated uploads. Without it, anyone who can reach the UploadThing route can upload files to the bucket. This is a pre-existing pattern (other endpoints like `questionImage` also lack middleware), but since this diff adds the profile page that relies on this endpoint, it's worth flagging.

## Dashboard rank computation

```typescript
const allUserScores = await db.testResult.groupBy({
  by: ['userId'],
  _sum: { score: true },
});

const usersWithHigherScore = allUserScores.filter(
  (u) => (u._sum.score || 0) > myScore
).length;
```

This loads one row per user who has ever taken a test into Node.js memory, then filters with Array.filter. At 100K users this becomes problematic per dashboard load. The computation could be a single `COUNT` query with a `HAVING` clause or a raw SQL `RANK()` window function.

## Teacher tests route: production logging

The `teacher/tests/route.ts` diff adds three `console.log` statements that log teacher IDs and user IDs on every request. These leak internal identifiers to whatever log aggregation is in place and add noise in production. Remove or gate behind `process.env.NODE_ENV === 'development'`.
