# YouTube player simplification, live dashboard stats, and profile management (v2)

All six issues from the first review pass have been addressed: debug endpoint deleted, image URL validation added (UploadThing/Google/Cloudinary allowlist), UploadThing avatar endpoint gated with auth middleware, rank query replaced with raw SQL `COUNT`, production `console.log` removed from teacher tests, and `jose` is now an explicit `^5.2.0` dependency in `package.json` to match the Telegram callback's usage of the v5 API.

**Watch for:** Streak calculation fetches all user results to build a date set in memory when a single SQL query would suffice (confirmed). Name field has no maximum length, allowing arbitrarily long strings into the database (confirmed).

**Verdict**: APPROVED

## High-level view

The YouTube player dropped from ~250 lines of IFrame API orchestration to ~130 lines of a straightforward `youtube-nocookie.com/embed` iframe with native controls, eliminating the Error 153 trigger.

The dashboard stats endpoint's streak calculation is the remaining performance concern: it fetches all `completedAt` timestamps for the user (up to 365 days) and builds a date set in JavaScript. A heavy test-taker accumulating thousands of results would benefit from a `DISTINCT DATE(completedAt)` query instead of fetching full rows.

The profile `PATCH` name field accepts any non-empty string without an upper bound on length — practically harmless for a small user base but a minor gap that could cause UI overflow or database bloat.

## Streak computation pulls full result set

The streak calculation in `/api/dashboard/stats` fetches every `completedAt` field for the user in the last 365 days:

```typescript
const results = await db.testResult.findMany({
  where: { userId, completedAt: { gte: startDate } },
  select: { completedAt: true },
  orderBy: { completedAt: 'desc' },
});
```

It then builds a `Set<string>` of date strings and walks backwards from today. A user with 2,000 test results sends 2,000 rows over the wire just to deduplicate into at most 365 date strings. The deduplication could be pushed to Postgres:

```sql
SELECT "completedAt"::date as d
FROM "TestResult"
WHERE "userId" = $1 AND "completedAt" >= $2
GROUP BY d
ORDER BY d DESC
```

This returns at most 365 rows regardless of how many tests the user has taken. The consecutive-day walk can remain in JS since it's bounded to 365 iterations.

## Name field unbounded length

The profile PATCH only checks `name.trim().length > 0`:

```typescript
if (typeof name === 'string' && name.trim().length > 0) {
  updateData.name = name.trim();
}
```

A client could submit a multi-megabyte name string. Prisma will pass it through to Postgres (the `name` column is `TEXT` with no constraint). Adding a cap (e.g., 100 characters) prevents bloat and keeps the UI predictable.
