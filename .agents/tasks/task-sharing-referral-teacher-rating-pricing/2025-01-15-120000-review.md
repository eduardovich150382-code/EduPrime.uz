# Social sharing, referral system, teacher CRUD, and test UI overhaul

This change adds four features to the education platform: a referral system with reward mechanics (5 friends = 5 days premium), social sharing of test results with a public share page, a fully functional teacher panel with real database-backed stats and test CRUD, and multiple UI/UX fixes (difficulty signal bars, cover images, category filtering, video player fallback, always-visible question options, real leaderboard, pricing deep links to Telegram bot).

The approach is broad but well-structured: new Prisma models (`Referral`, `coverImage` field on `Test`), a set of API routes for each feature domain, and client-side pages that call those APIs. The share page is made public via middleware. The Telegram bot webhook handles plan-specific `/start` params for pricing deep links.

**Watch for:** The referral link leads to `/login?ref=CODE` but the login page and auth callbacks never read or persist the `?ref=` param, so referrals will never actually be recorded (confirmed). The reward detection query (`paymentId: null` in check-reward vs `duration: 'ONE_MONTH'` in the GET endpoint) uses inconsistent heuristics that could false-positive on unrelated subscriptions (likely). The teacher stats revenue calculation equates "test results" with "purchases" which inflates revenue for anyone who took a paid test without paying (confirmed).

## High-level view

The referral system has the schema, the APIs, and the dashboard UI, but the critical integration point is missing: nothing in the Google OAuth or Telegram login flow reads the `?ref=` query parameter from the URL and calls `POST /api/referral`. The referral pipeline is disconnected at the signup boundary, so the `Referral` table will remain empty.

The public share page at `/share/[id]` works end-to-end: the middleware adds `/share` to publicPaths, the API fetches the result without auth, and the UI renders motivational text plus platform info. The share URL is constructed from `result.id` which is a cuid, so it's unguessable but not access-controlled — once shared, the link permanently exposes the user's name and image.

The teacher panel replaces hard-coded mock data with real queries. Stats, test list, edit page, publish/unpublish toggle, and delete all work through proper API routes with ownership verification. The question update strategy is delete-all-then-recreate, which loses question IDs between saves and may break historical result references.

The tests page category filtering sends a `type` param to `/api/tests` which cross-references `TestCategory.type`. Difficulty now uses signal bars instead of stars. Cover images show when present; gradient fallback is deterministic by array index. Tests without a category won't appear under any filter except "Barchasi".

The YouTube player adds a 3-second timeout before falling back to native iframe controls, addressing the case where the IFrame API script doesn't load. The pricing page constructs deep links like `buy_premium_1month` which the Telegram webhook parses and responds with payment card details.

<summary>Issues (6)</summary>

1. **Referral link dead end** — The referral URL `/login?ref=CODE` is never consumed. The login page and auth callbacks ignore the `?ref=` param, so referrals are never recorded. Add ref-param capture in the login page or auth signIn callback, then call `POST /api/referral` with the new user ID.
2. **Reward detection inconsistency** — `check-reward` looks for `paymentId: null` while the GET endpoint looks for `duration: 'ONE_MONTH'`. Any subscription granted through other means (admin, promo) could false-match. Use a dedicated flag or a `source` field on the subscription to identify referral rewards.
3. **Revenue equals attempts** — `/api/teacher/stats` multiplies test price by total result count, conflating "took the test" with "paid for the test". Revenue should be derived from actual Payment records tied to that test or teacher.
4. **Question delete-and-recreate** — The edit endpoint deletes all questions then recreates them. If `TestResult.answers` stores question IDs for rendering the results page, those references become stale after an edit.
5. **Share page exposes user info** — The public share API returns user name and image with no privacy opt-out. Once a link is shared publicly it becomes a permanent exposure of the test-taker's profile.
6. **Referral code collision** — `generateReferralCode` uses 6 random chars appended to a name slug. At scale, collisions are possible and the `@unique` constraint would throw an unhandled Prisma error. Add a retry loop or use a UUID.

## Referral pipeline is disconnected at signup

The referral link format is `/login?ref=CODE`. The dashboard page constructs it correctly. But the login page (`/[locale]/(auth)/login/page.tsx`) calls `signIn('google', { callbackUrl: '/dashboard' })` without ever reading `searchParams.ref`. The auth callbacks in `src/lib/auth.ts` don't receive or forward the referral code either. The `POST /api/referral` endpoint exists and correctly creates the `Referral` record, but nothing ever calls it during the signup flow.

The consequence: the `Referral` table stays empty, the progress bar on the referral page stays at zero, and no user ever earns the 5-day premium reward. This is the most critical gap in the change.

## Reward detection uses fragile heuristics

The `POST /api/referral/check-reward` endpoint identifies existing rewards by:

```typescript
const existingReward = await db.subscription.findFirst({
  where: {
    userId,
    plan: 'PREMIUM',
    isActive: true,
    paymentId: null, // Referral reward has no payment
  },
});
```

Meanwhile the `GET /api/referral` endpoint checks:

```typescript
const rewardSubscription = await db.subscription.findFirst({
  where: {
    userId,
    plan: 'PREMIUM',
    duration: 'ONE_MONTH',
  },
});
```

These two queries identify "referral reward" differently. A subscription created by an admin (also `paymentId: null`) would match the first. Any purchased one-month premium would match the second. A `source: 'REFERRAL'` field on the subscription model would make this unambiguous.

The granted subscription uses `duration: 'ONE_MONTH'` with a comment acknowledging it's actually 5 days. If the system checks `duration` for access expiration logic elsewhere, this mismatch could grant 30 days instead of 5.

## Teacher revenue is inflated

The stats endpoint calculates revenue as `test.price * resultCount` — every test attempt counts as a purchase. A user whose subscription grants access without paying per-test would inflate this number. Revenue should come from confirmed `Payment` records filtered by the teacher's tests.

## Question edit is destructive

```typescript
await db.question.deleteMany({ where: { testId: id } });
await db.question.createMany({ data: ... });
```

All question IDs change on every save. `TestResult.answers` likely stores question IDs as JSON references for rendering the results page. After an edit, historical results would fail to match answers to their questions. An upsert-by-order or stable-ID approach would preserve referential integrity.

## Share page privacy

The public share API at `/api/share/result/[id]` returns the test-taker's `name` and `image` to any unauthenticated visitor. Result IDs are cuids (25 chars, high entropy) so they're not guessable, but once a user shares their link on social media it becomes permanently accessible. There's no mechanism to revoke or expire a shared result.
