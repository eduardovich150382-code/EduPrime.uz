# Social sharing, referral system, teacher CRUD, and test UI overhaul (v2)

The four previously raised blocking issues have been addressed: the login page now captures `?ref=` into localStorage, a `ReferralHandler` component in the main layout calls `POST /api/referral` after authentication, the teacher stats endpoint returns paid test attempt counts instead of inflated revenue, the referral code generator retries on collision, and the reward detection in `GET /api/referral` now uses the same `paymentId: null + isActive: true` query as `check-reward`.

**Watch for:** The question edit endpoint still uses delete-all-then-recreate, which breaks question ID references stored in historical `TestResult.answers` (confirmed). The referral POST endpoint has no authentication — any caller can register a referral for any user ID pair by guessing a valid code and user ID (confirmed). The Telegram webhook payment confirmation flow stores pending payments but never actually creates a `Subscription` record in the database (confirmed).

## High-level view

The referral pipeline is now connected end-to-end. The login page persists the ref code, `ReferralHandler` fires after session authentication, and the API registers the referral. The retry-with-fallback approach for code generation eliminates the unique constraint crash. One gap remains: the `POST /api/referral` endpoint is unauthenticated, meaning anyone with a user ID and a referral code can fabricate referral records without actually signing up through the link.

The teacher panel is functional with real data and ownership-verified CRUD. The delete-all-recreate question strategy on the edit endpoint is the remaining structural concern — `TestResult.answers` stores question IDs as JSON, and those IDs become orphans after a teacher edits their test. Historical results would render correctly only if the result page can fall back gracefully when question IDs don't match.

The Telegram webhook handles pricing deep links (`buy_premium_1month` etc.) and shows payment card details to the user. However, the admin `confirm_` callback only sends a congratulatory message and deletes the pending payment record — it never creates a `Subscription` in the database. Users would receive a "your plan is activated" message but remain on the free tier.

<details>
<summary>Issues (4)</summary>

1. **Referral POST is unauthenticated** — `POST /api/referral` accepts `{ referralCode, newUserId }` without verifying the caller is the `newUserId`. Any client can call this endpoint with a crafted body to inflate someone's referral count. Add `auth()` and verify `session.user.id === newUserId`.
2. **Question delete-and-recreate breaks historical results** — `PUT /api/teacher/tests/[id]/questions` deletes all questions and creates new ones with fresh IDs. `TestResult.answers[].questionId` references become stale. If the results page joins on question ID to render correct/incorrect state, edited tests break all prior result views.
3. **Telegram payment confirmation never creates Subscription** — The `confirm_` callback in the webhook sends the user a success message and deletes the pending payment setting, but doesn't call `db.subscription.create(...)` or `db.payment.create(...)`. The user's actual access level is never upgraded.
4. **Share page exposes user info permanently** — The public share API returns user name and image to any visitor. No revocation or privacy opt-out exists. Once a result link is posted publicly, the profile data is permanently accessible.

## Referral POST endpoint lacks authentication

The `POST /api/referral` handler creates a `Referral` record without checking the session:

```typescript
export async function POST(request: NextRequest) {
  const body = await request.json();
  const { referralCode, newUserId } = body;
  // ... no auth() call, no session check
  await db.referral.create({
    data: { referrerId: referrer.id, referredUserId: newUserId },
  });
}
```

The endpoint is publicly callable. An attacker who knows any user's ID (cuids are high-entropy but visible in client state) and a valid referral code can POST fabricated referrals to push someone to the 5-referral reward threshold. The self-referral check only blocks `referrer.id === newUserId`, not fabrication of referrals from other user IDs the attacker controls or guesses.

Fix: add `const session = await auth()` and verify `session.user.id === newUserId`.

## Question update destroys referential integrity

The edit page saves questions through `PUT /api/teacher/tests/[id]/questions`:

```typescript
await db.question.deleteMany({ where: { testId: id } });
await db.question.createMany({ data: questions.map(...) });
```

`TestResult.answers` is stored as a JSON array with `questionId` references:
```typescript
answers: { questionId: string; answer: string; isCorrect: boolean }[]
```

After an edit, every `questionId` in every historical result for that test points to a deleted record. The results page fetches the test's current questions and matches them to the answer array by ID. Edits break all prior result views for that test — answers appear unmatched, and the user sees a broken results page.

An upsert approach (update existing questions by ID, only delete removed ones, create new ones) would preserve referential integrity for unmodified questions.

## Telegram webhook confirm flow is incomplete

The `confirm_` callback handler:

```typescript
if (data.startsWith('confirm_')) {
  const targetUserId = data.replace('confirm_', '');
  const paymentData = await db.systemSetting.findUnique({ ... });
  if (paymentData) {
    const payment = JSON.parse(paymentData.value);
    await sendMessage(parseInt(targetUserId), '🎉 Tabriklaymiz! ...');
    await db.systemSetting.delete({ where: { key: `pending_payment_${targetUserId}` } });
    await sendMessage(chatId, `✅ @${payment.username} uchun tarif aktivlashtirildi.`);
  }
}
```

No `Subscription` or `Payment` record is created. The user remains on the free tier despite receiving an "activated" message. The payment data in `SystemSetting` contains `plan`, `duration`, and `amount` — all needed to create the subscription — but the creation step is missing. After the admin confirms, the handler should create a `Payment` record and a `Subscription` with the appropriate `startDate`/`endDate` based on the selected duration.
